Structured binding

Do you do this often?


void g(T t) {
  auto x = t.x;
  auto y = t.y;
  auto z = t.z;
  // ...
}
The destructuring problem: how do we assign convenient names to the components of a tuple-like struct?

How about this?


void g(T& t) {
  auto& x = t.x;
  auto& y = t.y;
  auto& z = t.z;
  // ...
}
We may want those names to have reference semantics instead of value semantics, too!

Ain't this beautiful?


std::tuple<T1, T2, T3> f();

T1 x; T2 y; T3 z;
std::tie(x, y, z) = f();
Before C++17, the only way to do destructuring is with std::tie() Problems with this: The variables need to be separately declared before they are std::tie()‚Äôd.

Oops-prone...


std::tuple<T1, T2, T3> f();

int x; T2 y; T3 z;
std::tie(y, y, z) = f();
More problems: If the variables are PODs, they may be uninitialized. Note the bug in the std::tie() call here ‚Äì we‚Äôve passed y twice! We‚Äôll get no error for this, and x will be left uninitialized

What about:

  • Non-default-constructible types
  • Types we don't want to initialize (expensive?)
  • const
  • Types with reference members

auto [x, y, z] = obj;

üòÅ

The type of obj must be Destructurable:

  • Either all non-static data members:
    • Must be public
    • Must be direct members of the type or members of the same public base class of the type
    • Cannot be anonymous unions
  • Or the type has:
    • An obj.get<>() method or an ADL-able get<>(obj) overload
    • Specializations of std::tuple_size<> and std::tuple_element<>
STL destructurable types: array, tuple, pair.

Any why make_* all the things?

How does it work?

  • Simple declarations of variables (or variable templates) that are also definitions whose declarator is a noptr-declarator (i.e. not when declaring functions, template parameters, function parameters, non-static data members, pointers, references etc.), and
  • Explicit type conversion (functional notation)

Translated to English (when C++ is ‚â≤ English):


    template<class ... Ts> struct X { X(Ts...) };
    X x1{1}; // OK X<int>
    X x11; // OK X<>
    template<class T> X xv{(T*)0}; // OK decltype(xv<int>) == X<int*>
    extern X x2; // NOT OK, needs to be a definition
    X arr[10];   // OK X<>
    X x1{1}, x2{2}; // OK, deduced to the same type X<int>
    X *pointer = 0; // NOT OK
    X &&reference = X<int>{1};
    X function(); // NOT OK

Folding expressions


auto sum() { return 0; }

template <typename T>
auto sum(T&& t) { return t; }

template <typename T, typename... Rest>
auto sum(T&& t, Rest&&... r) {
  return t + sum(std::forward<Rest>(r)...);
}

Will it fold?

Unary Right(E op ...)‚â£E1 op (... op (EN-1 op EN))
Unary Left(... op E)‚â£((E1 op E2) op ...) op EN
Binary Right(E op ... op I)‚â£E1 op (... op (EN-1 op (EN op I)))
Binary Left(I op ... op E)‚â£(((I op E1) op E2) op ...) op EN

All binary operators are foldable


        ==  !=  <   >   <=  >=  &&  ||  ,   .*  ->*  = 
        +   -   *   /   %   ^   &   |   <<   >> 
        +=  -=  *=  /=  %=  ^=  &=  |=  <<=  >>=
For unary fold expressions, if the parameter packis empty then the value of the fold is: && ‚â£ true || ‚â£ false , ‚â£ void() For any operator not listed above, an unary fold expression with an empty parameter pack is ill-formed.

Execution policies:

  • std::seq indeterminately sequenced in the calling thread
  • std::par indeterminately sequenced with respect to each other within the same thread
  • std::par_unseq unsequenced with respect to each other and possibly interleaved
